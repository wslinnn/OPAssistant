# Product

## Register

product

## Platform

web

## Users

主要用户是**技术型玩家与极客**:自己刷 OpenWrt、熟悉 rpcd/ubus/SSH,把路由器当家域基础设施来折腾和运维的人。使用场景常常是**跨网远程**——人不在家、出差、或想从移动网络顺手调整家里或办公点的路由器。他们要完成的核心任务是:在手机上以原生 App 的体感,完成原本要开电脑进 luci 网页端才能做的配置与诊断,且改完不把网改断。这是一个单一明确的主受众,没有需要单独拆分的次受众。

## Product Purpose

OPAssistant 是 OpenWrt 路由器的移动端管理客户端,一套 uni-app 代码同时发行 Android 与 H5。它通过路由器已有的 rpcd ubus JSON-RPC 通道通信,**不要求路由器额外安装任何 luci 插件或自定义 ACL**,因此能适配绝大多数 OpenWrt 系统。功能覆盖设备总览(仪表盘 / 网络 / 无线客户端 / 路由表 / 流量统计)、十个插件的原生编辑(arpbind、防火墙、PassWall2 等)、以及一整套只读运维工具(ping / traceroute、系统日志、conntrack、进程、启动项)。它存在的意义是让远程运维成为日常可承受的操作,而非每次都得坐到路由器跟前。成功 = 用户跨网改防火墙、WiFi、代理配置时也敢放手下手,因为每一步写操作都有 rollback 兜底与分级确认兜着,失联会自动回滚自愈。

## Positioning

**像原生系统 App 一样管路由器**——不是把 luci 网页搬进手机壳,而是提供一种移动原生、系统级设置般自然的路由器管理体感。这是每一屏都要强化的唯一战略主张:交互、信息密度、节奏都向"第一方系统 App"看齐,而不是向"网页适配"妥协。

## Brand Personality

气质是**精准 · 冷静 · 可信赖**。语气专业、低噪、不煽情——像一个沉得住气的运维搭档:把事实和状态说清楚,把该确认的风险讲明白,然后退到一边让你决定。情绪目标是**信任感**:用户敢把跨网的、可能断网的操作交给它。这不是热情或惊艳,而是"用得越久越放心"。

## Anti-references

- **花哨 / 过度装饰**:渐变文字、滥用毛玻璃、大色块 hero、emoji 堆砌——工具类 App 的忌讳,出现即破坏信任。
- **luci 网页端的拥挤表格感**:密集表单、小字、低对比、字段堆叠——能做的事很多,但绝不能看起来像 luci 那面"墙"。
- **通用 SaaS 仪表盘模板**:hero-metric 大数字、千篇一律的等高卡片网格、营销味——这是后台模板,不是系统 App。

## Design Principles

- **像系统 App,不是网页包装。** 每一处交互、转场、信息密度都向移动端第一方系统设置看齐;任何"网页感"(密集表格、浏览器式控件)都是回归信号。
- **远程敢下手。** 安全不是功能,是前提。任何写路径都必须先挣得用户信任——rollback 兜底、分级确认、保存前 session 预检——把"失联自愈"做成默认而非选项。
- **克制即专业。** 装饰越少,越像懂行的人做的。让结构与状态本身承担表达,抑制一切为"好看"而生的视觉噪声。
- **一套语言,管十种插件。** 同一个 schema 编辑器、同一套 token、同一组交互模式贯穿全站;一致性优先于新鲜感,新增能力的第一选择是复用而非另起炉灶。
- **尊重专家,但不堆砌。** 用户是极客,要给真深度(防火墙 zone、PassWall2 分流、conntrack 全量),但用渐进式披露与清晰分层把它组织好——深而不乱,永远不变成 luci 那面表单墙。

## Accessibility & Inclusion

标准 · 意识型:在对比度(正文 ≥ 4.5:1)、触控热区、动效可关闭(`prefers-reduced-motion`)等基础项上做到位,把可读性和可操作性当作默认质量门槛,但不追求 WCAG AAA 级别的严格认证。远程与小屏场景下,信息一眼可读比合规条目更重要。
